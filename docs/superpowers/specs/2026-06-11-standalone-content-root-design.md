# SilentFall Blog `standalone` 内容根目录修复设计

## 背景

当前用户端 `SFBlogs` 使用 Next.js `standalone` 作为正式生产部署方式。实测发现：

- 管理端发布并同步文章成功；
- 新文章文件已经写入 `SFBlogs/posts`；
- 用户端使用 `node .next/standalone/server.js` 启动后，访问新文章返回 `404`；
- 用户端使用 `next start` 启动时，新文章又可以正常访问。

根因已经确认：`standalone` 启动入口会把运行时工作目录切换到 `.next/standalone`，导致代码中依赖 `process.cwd()` 的内容读取逻辑错误地读取构建产物目录，而不是实际博客内容目录。

## 目标

- 修复用户端在 `standalone` 生产运行时无法读取新同步内容的问题。
- 统一所有动态内容读取入口，避免只修文章页而遗留首页、时间线、说说页、关于页等同类问题。
- 保持开发环境、本地普通启动、生产 `standalone` 启动三种场景行为一致。
- 不改动后台同步协议，不引入“同步后必须重建前端”的额外运维要求。

## 非目标

- 不重构后台同步接口。
- 不改为“每次同步后复制内容到 `.next/standalone`”的补丁式方案。
- 不引入数据库化内容存储。
- 不调整现有评论、访客说说等 MongoDB 接口逻辑。

## 现状与影响范围

当前 `SFBlogs` 中至少以下页面或模块依赖 `process.cwd()` 读取动态内容：

- `app/page.tsx`
- `app/posts/page.tsx`
- `app/posts/[slug]/page.tsx`
- `app/timeline/page.tsx`
- `app/moments/page.tsx`
- `app/about/page.tsx`

这些页面分别读取：

- `posts/*.md`
- `moments/*.md`
- `app/about/about.md`

因此，如果只修文章详情页，首页文章流、时间线聚合、说说页面、关于页仍然可能在线上读取旧内容或直接缺失内容。

## 方案对比

### 方案一：统一内容根目录解析工具

做法：

- 在 `SFBlogs/lib` 下新增统一内容路径解析工具；
- 优先从环境变量读取博客真实内容目录；
- 未配置时自动识别当前是否运行于 `.next/standalone`；
- 若处于 `standalone`，则回退到真实项目根目录；
- 所有读取内容的页面统一改用该工具拼接路径。

优点：

- 修复面完整；
- 逻辑集中，后续维护成本低；
- 不依赖构建产物目录；
- 与后台同步目录 `SFBlogs` 自然对齐。

缺点：

- 需要修改多个读取入口；
- 需要补一层路径解析抽象。

### 方案二：同步时额外复制到 `.next/standalone`

做法：

- 后台同步时同时把内容复制到 `.next/standalone/posts`、`.next/standalone/moments` 等目录。

优点：

- 表面上改动较少。

缺点：

- 强耦合构建产物；
- 每次重新构建都可能覆盖；
- 容易与部署脚本、发布流程互相打架；
- 本质上没有解决路径解析错误。

### 方案三：同步后强制重建并重启前端

做法：

- 后台同步后要求重新构建或重启前端。

优点：

- 技术实现最简单。

缺点：

- 失去实时同步价值；
- 运维成本高；
- 不符合当前项目“后台更新后用户端尽快可见”的目标。

## 选型

采用方案一：统一内容根目录解析工具。

这是唯一兼顾正确性、可维护性和线上可用性的方案。

## 设计

### 1. 新增内容根目录解析工具

新增一个工具模块，职责包括：

- 返回真实博客内容根目录；
- 提供文章目录、说说目录、关于页内容文件等辅助路径；
- 兼容以下三种场景：
  - 开发态直接从项目根目录运行；
  - 普通生产启动；
  - `standalone` 生产启动。

解析策略按优先级执行：

1. 若环境变量显式指定内容根目录，则优先使用；
2. 否则基于当前文件所在位置推断项目根目录；
3. 若运行路径位于 `.next/standalone`，则自动跳出该目录，定位到真实 `SFBlogs` 根目录；
4. 若最终目录缺失 `package.json` 等关键标记，则保守回退并输出明确错误。

### 2. 替换所有内容读取入口

把原来直接使用 `process.cwd()` 的页面改为统一调用内容路径工具：

- 首页读取最近文章与说说；
- 文章列表读取文章目录；
- 文章详情读取指定文章；
- 时间线读取文章与说说；
- 说说页读取说说目录；
- 关于页读取 `about.md`。

### 3. 回归测试策略

先增加会失败的测试，再实现修复。

测试分为两层：

- 路径解析单元测试：
  - 模拟普通根目录运行；
  - 模拟 `.next/standalone` 运行；
  - 确认解析结果都指向真实博客根目录。
- 生产回归测试：
  - 重新构建用户端；
  - 用 `standalone` 启动；
  - 通过管理端发布同步新文章；
  - 立即访问新文章页面；
  - 再检查首页或列表中是否能看到新内容。

### 4. 错误处理

- 若真实内容目录无法解析，页面读取逻辑应抛出明确错误或进入已有的 `notFound()` 分支，而不是静默读取错误目录。
- 若目录存在但文件不存在，仍保持当前 `notFound()` 语义。
- 若环境变量配置了非法目录，优先暴露配置错误，避免悄悄回退到错误路径。

## 风险

- 多页面同时切换读取方式，若工具函数设计不稳，可能造成内容页全面异常。
- 若路径工具过度依赖运行目录字符串匹配，后续部署结构变化时可能再次失效。

## 降险措施

- 把根目录解析逻辑集中在一个工具文件，不在页面中复制判断逻辑。
- 优先使用稳定锚点推断项目根目录，而不是依赖易变的命令启动目录。
- 通过生产回归测试验证“发布后立即访问”而不是只验证构建通过。

## 验收标准

- `SFBlogs` 重新构建成功。
- 用户端使用 `node .next/standalone/server.js` 启动成功。
- 管理端发布同步新文章后，`SFBlogs/posts` 出现新文件。
- 无需重建用户端，即可访问新文章页面且返回 `200`。
- 首页、文章列表、时间线、说说页、关于页不因本次改动出现读取异常。
