# 如何通过宝塔面板部署 SilentFall_Blog

本文档基于当前代码库的真实实现整理，适用于将 `SFBlogs`、`my-blog-manager` 与 `my-blog-manager/cms_core` 部署到 Linux 服务器。文中默认示例域名为 `blog.your_domain.com` 与 `admin.your_domain.com`，请按实际环境替换。

> 前提条件：一台已联网的 Linux 服务器，建议 Ubuntu 22.04 或 Debian 12，配置建议不低于 2 核 2G。

## 1. 服务器准备与宝塔面板安装

### 1.1 通过 SSH 连接服务器

第一步，使用 SSH 登录服务器：

```bash
ssh root@your_server_ip
```

如果是首次连接，输入 `yes` 确认指纹，然后输入服务器密码。

### 1.2 安装宝塔面板

第二步，根据系统执行宝塔安装命令。

Ubuntu / Debian：

```bash
wget -O install.sh https://download.bt.cn/install/install-ubuntu_6.0.sh && bash install.sh ed8484bec
```

CentOS：

```bash
yum install -y wget && wget -O install.sh https://download.bt.cn/install/install_6.0.sh && bash install.sh ed8484bec
```

安装完成后，保存面板地址、用户名与密码。

## 2. 登录宝塔面板并安装基础软件

### 2.1 登录面板

第一步，浏览器打开宝塔面板地址。

第二步，输入安装完成后得到的账号密码登录。

第三步，按提示完成面板初始化。

### 2.2 安装 LNMP 套件

本项目只需要 Nginx，不依赖 PHP 与 MySQL。推荐安装：

- Nginx：最新版稳定版
- PHP：不安装
- MySQL：不安装

如果你已经手动安装了 Nginx，也可以跳过这一节。

## 3. 安装运行环境（Node.js / Python / MongoDB / PM2）

### 3.1 安装 Node.js

第一步，进入宝塔软件商店并安装“Node.js 版本管理器”。

第二步，安装 Node.js 18 或以上版本。

第三步，在终端验证：

```bash
node -v
npm -v
```

### 3.2 安装 Python

第一步，进入宝塔软件商店安装“Python 管理器”。

第二步，安装 Python 3.10 或以上版本。

第三步，在终端验证：

```bash
python3 --version
pip3 --version
```

### 3.3 安装 MongoDB

第一步，进入宝塔软件商店安装 MongoDB。

第二步，确认 MongoDB 服务状态为“运行中”。

### 3.4 安装 PM2

在终端执行：

```bash
npm install -g pm2
pm2 -v
```

## 4. 上传项目文件

### 4.1 创建项目目录

第一步，在宝塔文件管理器进入 `/www/wwwroot`。

第二步，新建项目目录，例如：

```text
/www/wwwroot/SilentFall_Blog
```

### 4.2 上传方式

你可以选择以下任一方式：

方式一，本地压缩后上传。

方式二，服务器内直接克隆：

```bash
cd /www/wwwroot/SilentFall_Blog
git clone https://github.com/your_name/your_repo.git .
```

### 4.3 确认目录结构

确保最终目录结构如下：

```text
/www/wwwroot/SilentFall_Blog/
├── SFBlogs/
├── my-blog-manager/
├── README.md
└── 如何用服务器的宝塔面板将博客项目部署到你的服务器中.md
```

注意：`SFBlogs` 和 `my-blog-manager` 必须是同级目录。当前代码在未显式配置同步路径时，会自动尝试识别同级的 `SFBlogs` 目录。

## 7. 配置 MongoDB 数据库

### 7.1 确认 MongoDB 运行

第一步，在宝塔软件商店确认 MongoDB 已启动。

第二步，命令行验证服务状态：

```bash
mongosh --eval "db.runCommand({ ping: 1 })"
```

### 7.2 验证连接

当前代码默认连接：

- 连接串：`mongodb://localhost:27017`
- 数据库名：`silentfall_blog`

你也可以改为用户名密码模式，例如：

```text
mongodb://blog_user:your_password@127.0.0.1:27017/silentfall_blog?authSource=admin
```

建议在生产环境创建专用数据库用户并授予读写权限。

### 7.3 无需额外配置

本项目没有固定的 SQL 初始化脚本，首次写入时会自动创建数据库和集合。若你需要迁移历史数据，可使用：

```bash
mongorestore --uri="mongodb://blog_user:your_password@127.0.0.1:27017" /path/to/backup
```

## 8. 配置安全环境变量

### 8.1 生成 JWT 密钥

第一步，生成随机密钥：

```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
```

### 8.1 生成密码哈希和管理密钥

第二步，生成后台密码哈希：

```bash
python3 -c "import bcrypt; print(bcrypt.hashpw(b'your_admin_password', bcrypt.gensalt()).decode())"
```

### 8.2 创建环境变量文件

第三步，在 `my-blog-manager/.env` 中写入以下内容：

```dotenv
# MongoDB 配置
MONGO_URI=mongodb://blog_user:your_password@127.0.0.1:27017/silentfall_blog?authSource=admin
MONGO_DB_NAME=silentfall_blog

# 后台登录认证
CMS_ADMIN_USERNAME=admin
CMS_ADMIN_PASSWORD_HASH=your_bcrypt_hash
CMS_SECRET_KEY=your_secret_key
CMS_TOKEN_EXPIRE_HOURS=24

# 跨域白名单
CMS_ALLOWED_ORIGINS=https://blog.your_domain.com,https://admin.your_domain.com

# 可选：明确指定用户端物理路径
BLOG_FRONTEND_PATH=/www/wwwroot/SilentFall_Blog/SFBlogs

# 可选：如果希望 Next.js 代理直接使用完整地址
CMS_BACKEND_URL=http://127.0.0.1:8765

# 可选：如果不写 CMS_BACKEND_URL，则 Next.js 代理默认走同域 /cms-api
CMS_BACKEND_BASE_PATH=/cms-api
```

说明：

- 若 `SFBlogs` 与 `my-blog-manager` 已经同级放置，且 `my-blog-manager/data/deploy_config.json` 未配置路径，代码会自动推断目标目录。
- 若你的目录结构不同，建议明确设置 `BLOG_FRONTEND_PATH` 或在管理端设置页中保存同步目标路径。

## 9. 使用 PM2 启动后端服务

### 9.1 创建启动脚本

第一步，在 `/www/wwwroot/SilentFall_Blog/my-blog-manager` 下创建 `start-api.sh`：

```bash
#!/bin/bash
set -a
source /www/wwwroot/SilentFall_Blog/my-blog-manager/.env
set +a

cd /www/wwwroot/SilentFall_Blog/my-blog-manager
python3 -m uvicorn cms_core.main:app --host 127.0.0.1 --port 8765
```

赋予执行权限：

```bash
chmod +x /www/wwwroot/SilentFall_Blog/my-blog-manager/start-api.sh
```

### 9.2 启动后端

第二步，使用 PM2 启动：

```bash
cd /www/wwwroot/SilentFall_Blog/my-blog-manager
pm2 start ./start-api.sh --name silentfall-api
```

### 9.3 验证

第三步，验证 FastAPI 是否可用：

```bash
curl http://127.0.0.1:8765/api/status
```

预期返回示例：

```json
{"status":"online","message":"中枢神经已连接"}
```

## 12. 配置 Nginx 反向代理

### 12.1 添加站点

第一步，在宝塔面板中分别创建两个站点：

- `blog.your_domain.com`
- `admin.your_domain.com`

站点根目录可以任意，例如：

```text
/www/wwwroot/blog.your_domain.com
/www/wwwroot/admin.your_domain.com
```

### 12.2 配置反向代理

第二步，为用户端站点配置反向代理到 `3000`：

```nginx
location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

第三步，为管理端站点配置反向代理到 `3001`：

```nginx
location / {
    proxy_pass http://127.0.0.1:3001;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

### 12.3 配置 API 代理

第四步，为两个站点都增加 `/cms-api/` 代理。用户端站点与管理端站点都需要加：

```nginx
location /cms-api/ {
    proxy_pass http://127.0.0.1:8765/api/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

这样一来，当前代码在未显式设置 `CMS_BACKEND_URL` 时，也能自动通过同域 `/cms-api/` 访问 FastAPI。

### 12.4 配置后台管理器代理（推荐）

如果你希望管理端与用户端共用一个主域名，也可以让管理端挂到二级路径，例如 `/admin/`。但当前项目更推荐独立二级域名，便于鉴权、日志与缓存隔离。

## 18. 常见错误与解决方案

### 18.1 前端页面空白或样式丢失

原因通常是 `standalone` 目录缺少 `.next/static` 或 `public`。

处理方式：重新执行以下命令。

```bash
cp -r .next/static .next/standalone/.next/
cp -r public .next/standalone/
```

### 18.2 后端启动失败

常见原因：

- `.env` 中缺少 `CMS_ADMIN_PASSWORD_HASH`
- MongoDB 未启动
- Python 依赖未安装完整
- `MONGO_URI` 账号密码错误

排查命令：

```bash
pm2 logs silentfall-api --lines 100
```

### 18.3 构建时内存不足（OOM）

可以在构建前设置 Node.js 内存上限：

```bash
export NODE_OPTIONS=--max-old-space-size=4096
npm run build
```

### 18.4 评论或访客说说无法提交

优先检查以下四项：

1. `curl https://blog.your_domain.com/cms-api/status` 是否正常。
2. `CMS_ALLOWED_ORIGINS` 是否包含用户端与管理端实际域名。
3. MongoDB 是否运行，且 `MONGO_URI` 账号具有读写权限。
4. Nginx 是否把 `/cms-api/` 正确代理到了 `127.0.0.1:8765/api/`。

### 18.5 文章发布后前台不显示

优先检查以下四项：

1. `SFBlogs` 与 `my-blog-manager` 是否为同级目录。
2. `BLOG_FRONTEND_PATH` 或 `deploy_config.json` 中的 `blogPath` 是否正确。
3. 目标目录是否存在 `package.json`。
4. 发布后是否成功执行同步接口。

### 18.6 端口被占用

查看占用进程：

```bash
ss -lntp | grep -E '3000|3001|8765'
```

然后停止冲突进程或调整 PM2 配置。

### 18.7 Nginx 反向代理 502 错误

原因通常是目标进程未启动，或代理端口写错。

排查顺序：

1. `pm2 list`
2. `curl http://127.0.0.1:3000`
3. `curl http://127.0.0.1:3001`
4. `curl http://127.0.0.1:8765/api/status`

### 18.8 后台管理器登录后 Token 频繁失效

优先检查：

- `CMS_SECRET_KEY` 是否每次部署都变化
- 是否在多台机器间使用了不同的 `CMS_SECRET_KEY`
- 浏览器与服务器时间是否严重偏差

### 18.9 CORS 跨域错误

当前代码会根据 `CMS_ALLOWED_ORIGINS` 生成白名单。生产环境不要继续使用 `*`。正确示例：

```dotenv
CMS_ALLOWED_ORIGINS=https://blog.your_domain.com,https://admin.your_domain.com
```

如果仍然报错，请检查请求实际来源、协议和端口是否与白名单完全一致。
